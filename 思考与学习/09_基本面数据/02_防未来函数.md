# 防未来函数: 报告期 vs 披露日

> 整个 09 章最重要的一篇。
> 不懂这个, 你的基本面回测会"看起来很美, 上线就崩"。

## 一、5 秒钟理解

> 报告期 ≠ 披露日。
> 你拿到一份财报, 不是在报告期那一天, 而是 1~3 个月后。

例:
- 报告期 2024-03-31 (一季报截止日)
- 披露日 2024-04-25 左右 (公司实际公布的日期)
- → 2024-04-01 ~ 2024-04-24 这段时间, 你看不到这份一季报

如果你的回测里 2024-04-15 就用了 2024-03-31 的数据, 这就是赤裸裸的未来函数。

## 二、为什么这是致命的

一个典型的"看起来美" 例子:

```python
# 错误版本: 直接用报告期当时间索引
roe = quarterly_roe.reindex(daily_dates).ffill()
# 2024-03-31 这一天, roe 已经包含 2024 Q1 数据
# 但实际上 2024-04-15 之前没人能看到 Q1 数据!
```

回测时这样写, 你会发现:
- "选 ROE 增长最快的票" → 夏普 3.0!!
- 上实盘 → 夏普 0.3, 巨亏

为什么? 因为回测里你在用"未来 1~3 个月披露的财报" 来决定"今天买什么"。 这相当于穿越到未来读报纸再决定昨天买股票, 当然神准, 但实盘没法做到。

## 三、披露日规则 (中国 A 股)

证监会有强制披露时间表:

| 报告 | 报告期 | 强制披露截止 |
|---|---|---|
| 一季报 | 03-31 | 4 月 30 日前 |
| 半年报 | 06-30 | 8 月 31 日前 |
| 三季报 | 09-30 | 10 月 31 日前 |
| 年报 | 12-31 | 次年 4 月 30 日前 |

大部分公司最后一两天才披露 (不愿意提前暴露业绩), 所以披露日 ≈ 报告期 + 1~4 个月。

业绩好的公司会提前披露 (好消息要早讲), 差的拖到最后一天 ("反正都要披露, 让股价多坚持几天")。

## 四、本框架的近似规则

为了简化教学, 本框架使用:

```
披露日 = 报告期 + 45 天 (默认)
```

这意味着:
- Q1 (报告期 03-31) → 披露日 ≈ 05-15
- Q2 (06-30) → 披露日 ≈ 08-14
- Q3 (09-30) → 披露日 ≈ 11-14
- 年报 (12-31) → 披露日 ≈ 次年 02-14

45 天的选择是个中位数估计, 不完全准确 (有的公司更早披露, 有的恰好在监管 deadline)。 实战中更严谨的做法是从 `stock_yjbb_em` 接口拿真实披露日, 每个公司每个季度都对得上。

本框架的 `get_fundamental_factor` 接口支持调整:
```python
roe = get_fundamental_factor("roe", ..., announce_lag_days=60)  # 用 60 天近似
```

## 五、对齐到日频的核心代码

`src/data/fundamentals.py:to_daily` 干的就是这件事:

```python
def to_daily(bundles, field, trading_dates, announce_lag_days=45):
    for sym, bundle in bundles.items():
        q = bundle.quarterly[field].dropna()
        # 关键 1: 每条季报的"可用日" = 报告期 + 45 天
        available = pd.Series(
            q.values,
            index=q.index + pd.Timedelta(days=announce_lag_days),
        ).sort_index()

        # 关键 2: 在每个交易日, 用 ffill 取 "可用日 ≤ 当日" 的最近一条
        aligned = available.reindex(...).ffill().reindex(trading_dates)
```

效果: 在第 t 天, 因子值 = 最近一次"可用日 ≤ t" 的报告对应字段。

## 七、容易忽视的 4 个陷阱

### 陷阱 1: ffill 跨"假期" 还是用旧数据
正确的: 披露日之前一直用上一份报告。 错误的: 用下一份报告"提前 ffill" 回去。
代码层面就是 `pd.Series.ffill()` 而不是 `bfill()`。

### 陷阱 2: 多次披露调整
某些公司会在初次披露后做"重述" (Restatement), 给出修正后的数字。
实战中应该用初次披露版, 因为重述时回测时间点你拿不到修正版。 但很多接口只给"当前修订版"。
本框架默认用接口返回的最新值 (不区分初次/修订), 是一个简化, 教学层面可接受。

### 陷阱 3: 业绩预告先于正式财报
A 股有"业绩预告" 制度: 净利润可能提前 1~2 个月由公司预告。
严格防未来函数的做法应该把"业绩预告日" 也作为披露日。 这是高级技巧, 本框架未实现。

### 陷阱 4: 节假日 / 停牌
披露日可能是周末或假期, 这时数据要 ffill 到下一个交易日才能用。
本框架因为是 `trading_dates` 上 ffill, 自动正确处理。

## 八、自查清单: 你的基本面回测有没有用未来数据

跑完基本面回测前, 必须问自己:

```
[ ] 我用的数据时间戳是"报告期" 还是"披露日"?
[ ] 如果是报告期, 我加了 announce_lag_days 之类的偏移吗?
[ ] 第一份报告披露前的日子是 NaN, 还是被某种"默认值"填充了?
[ ] 我有没有用到"业绩预告" 数据? 如果有, 它的"预告日" 对齐了吗?
[ ] 修订版数据是不是不小心用进来了?
```

任一项不确定就停下查代码, 不要急着上回测。

## 九、跟价格数据的对比

为了让你建立直觉, 来看价格 vs 基本面在防未来函数上的差异:

```
价格数据 (日频, 易处理):
  2024-04-15 收盘价 → 当天就能用 (盘后)
  → 回测里默认 shift(1), 用前日收盘做今日决策, 简单粗暴

基本面数据 (季频, 难处理):
  2024-Q1 报告 → 4-30 之前才能拿到
  → 必须知道每条数据的"可用日", ffill 时严格控制
```

价格的"未来函数"很容易避免 (统一 shift(1) 即可), 基本面的"未来函数" 没法用单一规则避免, 必须每条数据各自对齐。

## 十、小结

- 报告期 ≠ 披露日, 这是基本面回测最容易爆雷的地方
- A 股披露日规则: 报告期后 1~4 个月
- 本框架近似: 披露日 = 报告期 + 45 天, 可调
- `to_daily` 函数自动按披露日对齐 + ffill 到日频
- 业绩预告 / 重述 / 假期等是更细节的陷阱, 本框架未全部实现
- 每次写基本面回测都过一遍 8 节的自查清单
